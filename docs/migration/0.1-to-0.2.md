# Migrating from Deep View v0.1 to v0.2

!!! info "Zero breaking changes"
    Every v0.1 invocation — CLI commands, Python API calls, plugin code,
    test fixtures, and config files — continues to work unmodified on
    v0.2. This guide walks through the *new* surface you can adopt and
    the handful of parser bug-fixes that ship alongside it.

## 1. Summary

Deep View v0.2 is a large, coordinated release that lands four new
subsystems next to the original memory / tracing / instrumentation / VM
stack. It adds **four new subpackages** (storage stack, offload engine,
container unlock, remote acquisition), **six new CLI command groups**,
**thirteen new optional-dependency extras**, **ten new typed event
classes**, **four new `AnalysisContext` lazy attributes**, and **four
new `DeepViewConfig` sub-trees**. It introduces **zero breaking
changes** — no removed APIs, no renamed CLI commands, no on-disk format
changes, no config knob default flips. Crashdump, hibernation, and
format-detection code paths that were previously stubs or fragile now
parse real-world inputs.

## 2. What's added

### 2.1 New subpackages

| Subpackage | Responsibility |
|---|---|
| `src/deepview/storage/` | Layered filesystem / ECC / FTL / encoding / partition / container pipeline driven by `StorageManager` |
| `src/deepview/offload/` | CPU + GPU job dispatch engine (`OffloadEngine`, pluggable `OffloadBackend`s) |
| `src/deepview/storage/containers/` | `UnlockOrchestrator` auto-unlock pipeline + `KeySource` hierarchy + `DecryptedVolumeLayer` |
| `src/deepview/memory/acquisition/remote/` | Remote memory-acquisition providers (SSH-DD, TCP, agent, LiME, IPMI, Intel AMT, DMA over Thunderbolt / PCIe / FireWire) with authorization gates |

Deep dives live under [architecture/storage](../architecture/storage.md),
[architecture/offload](../architecture/offload.md),
[architecture/containers](../architecture/containers.md), and
[architecture/remote-acquisition](../architecture/remote-acquisition.md).

### 2.2 New CLI command groups

| Command | Purpose |
|---|---|
| `deepview storage {list,info,wrap,mount}` | Inspect registered storage layers, probe filesystems, wrap raw NAND dumps with ECC + FTL, open a filesystem on a backing layer |
| `deepview filesystem {ls,cat,stat,find}` | Inspect a mounted filesystem layer |
| `deepview unlock {luks,auto,veracrypt,truecrypt}` | Pure-Python + libbde-backed unlock pipeline |
| `deepview unlock-native {bitlocker,filevault}` | Native-adapter unlock path deferring to libbde / libfvde |
| `deepview offload {status,run,benchmark}` | Inspect offload backends, submit ad-hoc jobs, compare KDF throughput |
| `deepview remote-image {ssh,tcp,agent,lime,ipmi,amt,dma-tb,dma-pcie,dma-fw}` | Authorization-gated remote memory capture, one subcommand per transport |

All new groups are listed in the full [CLI reference](../reference/cli.md).

### 2.3 New built-in plugins

New `@register_plugin` built-ins cover storage-adapter probes, container
detection, offload-job smoke tests, and remote-acquisition dry runs. All
are reachable from `plugins/builtin/__init__.py`, so any v0.1 code path
that iterates `context.plugins.list()` automatically sees them.

### 2.4 New `DumpFormat` enum values

| Value | Dump type |
|---|---|
| `DumpFormat.HIBERFIL` | Windows hibernation files (`hiberfil.sys`) |
| `DumpFormat.MINIDUMP_FULL` | Full minidump (MDMP with FullMemory flag) |
| `DumpFormat.VMWARE_VMEM` | VMware suspended-VM memory sidecar |
| `DumpFormat.VIRTUALBOX_SAV` | VirtualBox saved-state blob |
| `DumpFormat.HYPERV_VMRS` | Hyper-V saved-state blob |

Existing `DumpFormat` members keep their values and Python identities —
existing code comparing against `DumpFormat.RAW` or `DumpFormat.LIME`
continues to work unchanged.

### 2.5 New typed event classes

Ten new event classes published on `context.events` by the new
subsystems — see [reference/events](../reference/events.md) for field
schemas:

- `ContainerUnlockStartedEvent`
- `ContainerUnlockProgressEvent`
- `ContainerUnlockedEvent`
- `ContainerUnlockFailedEvent`
- `RemoteAcquisitionStartedEvent`
- `RemoteAcquisitionProgressEvent`
- `RemoteAcquisitionCompletedEvent`
- `OffloadJobSubmittedEvent`
- `OffloadJobProgressEvent`
- `OffloadJobCompletedEvent`

Event subscribers written for v0.1 events (`PluginStartedEvent`,
`MemoryDumpLoadedEvent`, `TraceEventReceivedEvent`, …) are unaffected.

### 2.6 New optional-dependency extras

Thirteen new extras declared in `pyproject.toml`:

| Extra | Packages | Enables |
|---|---|---|
| `storage` | `pytsk3`, `pyfsapfs`, `pyfsntfs`, `pyfsxfs`, `pyfsbtrfs`, `pyfsf2fs`, `pyfshfs`, `pyfsext` | Native filesystem adapters (read-only) |
| `compression` | `zstandard`, `lz4`, `python-lzo` | Memory-page compression decoders |
| `ecc` | `reedsolo`, `galois` | Reed-Solomon and BCH ECC codecs |
| `offload_gpu` | `pyopencl`, `pycuda` | OpenCL + CUDA offload backends |
| `containers` | `cryptography`, `argon2-cffi`, `pycryptsetup` (Linux), `libbde-python`, `libfvde-python` | LUKS / BitLocker / FileVault / VeraCrypt unlock |
| `remote_acquisition` | `paramiko`, `grpcio`, `grpcio-tools`, `python-ipmi`, `forensic1394` | Every remote transport |
| `docs` | `mkdocs`, `mkdocs-material`, `mkdocs-mermaid2-plugin`, `pymdown-extensions`, `mkdocs-asciinema-player-plugin` | Building this doc site locally |

The `all` aggregator is extended to cover every new extra; plain
`pip install -e ".[all]"` grabs everything.

### 2.7 New interfaces / ABCs

New abstract base classes under `interfaces/` (or local to the relevant
subpackage) that plugin authors can extend:

- `Filesystem` — read-only filesystem adapter contract
- `FTLTranslator` — flash-translation-layer contract
- `ECCDecoder` — error-correction-code decoder contract
- `OffloadBackend` — compute backend contract (extended by thread /
  process / OpenCL / CUDA / remote backends)
- `Unlocker` — encrypted-container unlock adapter contract
- `KeySource` — abstract key derivation source (extended by
  `MasterKey`, `Passphrase`, `Keyfile`)
- `RemoteAcquisitionProvider` — remote memory acquisition contract
  extending the existing `MemoryAcquisitionProvider`

### 2.8 New `AnalysisContext` lazy attributes

Four additions, all constructed on first access and cached:

| Attribute | Type | Notes |
|---|---|---|
| `context.offload` | `OffloadEngine` | Auto-registers `thread` + `process`; probes `gpu-opencl`, `gpu-cuda`, and `remote` stubs |
| `context.storage` | `StorageManager` | Auto-registers every in-tree filesystem / FTL / ECC adapter |
| `context.unlocker` | `UnlockOrchestrator` | Auto-discovers `UNLOCKER` exports from every container adapter module |
| `context.remote` | remote provider factory wrapper | Dispatches `factory.build_remote_provider(transport_name, endpoint)` |

The pre-existing attributes — `config`, `layers`, `events`, `platform`,
`artifacts`, `plugins` — keep their original construction semantics and
types. v0.1 test fixtures never touched the new attributes, so nothing
in the fixture tree needs adjusting.

### 2.9 New `DeepViewConfig` sub-configs

Four new sub-trees on `DeepViewConfig`, each following the existing
`DEEPVIEW_<SECTION>_<FIELD>` env-var override convention:

- `config.storage`
- `config.offload`
- `config.containers`
- `config.remote_acquisition`

All fields default to values that reproduce v0.1 behavior (i.e. when the
corresponding subsystem is not invoked, these knobs are inert).

## 3. Recommended install changes

!!! tip "No install changes required"
    If your v0.1 install already works, it still works. The invocations
    below are **additive opt-ins** for the new subsystems.

=== "Before v0.2"

    ```bash
    # Core install
    pip install -e ".[dev]"

    # With memory + instrumentation extras
    pip install -e ".[dev,memory,instrumentation]"

    # Everything
    pip install -e ".[all,dev]"
    ```

=== "After v0.2"

    ```bash
    # Core install — unchanged
    pip install -e ".[dev]"

    # With memory + instrumentation extras — unchanged
    pip install -e ".[dev,memory,instrumentation]"

    # Opt in to the new storage pipeline
    pip install -e ".[dev,storage,compression,ecc]"

    # Opt in to container unlock + GPU offload
    pip install -e ".[dev,containers,offload_gpu]"

    # Opt in to remote acquisition
    pip install -e ".[dev,remote_acquisition]"

    # Everything — unchanged invocation, now also pulls in the new extras
    pip install -e ".[all,dev]"
    ```

!!! warning "Heavy native deps stay optional"
    None of the new extras are pulled in by a bare `pip install -e .`.
    CI images that previously ran `pip install -e ".[dev]"` without the
    new extras will keep passing — tests that need a new extra are
    gated on the import and skipped when absent.

## 4. Hibernation and crashdump now actually parse

In v0.1 the hibernation and crashdump parsers were header-only stubs —
they recognised the magic bytes but raised on body reads. v0.2 ships
real parsers:

- **Hibernation** now decompresses Xpress-compressed page runs end to
  end. Code that previously wrapped `HibernationLayer.read()` in a
  try/except to tolerate the stub failure can drop the guard.
- **Crashdump** now handles the `BITMAP_DUMP` variant used by modern
  Windows kernels. Dumps that previously raised on the bitmap header
  now parse cleanly.

!!! warning "Behavior shift in error paths"
    If your v0.1 code relied on catching a specific exception from
    these stubs to route around them (for example, falling back to a
    raw read on any failure), that exception will no longer fire on
    well-formed inputs. Dumps that were silently triggering your
    fallback may now parse successfully and return different bytes
    — which is the correct behavior, but worth auditing.

See [architecture/storage](../architecture/storage.md) for the full
parser pipeline and
[guides/storage-image-walkthrough](../guides/storage-image-walkthrough.md)
for an end-to-end example.

## 5. `MemoryManager.detect_format()` is more aggressive

Detection in v0.1 keyed purely off magic bytes and a short allow-list.
v0.2 extends detection in three ways:

1. **New magic-byte recognition** — MDMP with the FullMemory flag,
   `hibr*` / `wake` hibernation headers, and SSM (Hyper-V saved state)
   blobs are now recognised directly.
2. **Extension fallback** — when magic bytes are ambiguous (empty
   prefix, zero-padded, or matching multiple candidates), detection
   consults the file extension (`.vmem`, `.sav`, `.vmrs`, `.dmp`, …)
   before giving up.
3. **Zero-padding bug-fix** — zero-padded raw images are no longer
   misclassified as ELF cores.

!!! warning "Audit your v0.1 detection-dependent code"
    If you pass files to `MemoryManager.detect_format()` whose format
    you already know, the return value is now more likely to be the
    *specific* format (`HIBERFIL`, `MINIDUMP_FULL`, `VMWARE_VMEM`, …)
    instead of the generic `RAW`. Code that matched on `DumpFormat.RAW`
    as a catch-all should either add the new members to its match
    arms or switch to a default branch.

Code that passes the format in explicitly (`MemoryManager.load(path,
format=DumpFormat.RAW)`) is unaffected — the detector is only consulted
when the format argument is omitted.

## 6. `AnalysisContext.for_testing()` is unchanged

!!! tip "Your test fixtures still work"
    The `for_testing()` class method keeps its v0.1 signature and
    semantics. It does not construct the new lazy attributes
    (`offload`, `storage`, `unlocker`, `remote`) unless your test
    touches them — just like `plugins` in v0.1, these are
    `@cached_property`-style attributes materialised on first access.

Tests built on the shared `context` fixture in
`tests/conftest.py` (`context = AnalysisContext.for_testing()`) continue
to run without modification. If a test does touch a new attribute, it
will construct a real engine; stub the relevant factory (for example,
patch `deepview.offload.engine.OffloadEngine`) if you need to isolate.

## 7. Adopting the new subsystems

### 7.1 Mount a filesystem on a disk image

```python
from deepview.core.context import AnalysisContext
from deepview.interfaces.layer import FileLayer

ctx = AnalysisContext()
layer = FileLayer("/path/to/disk.img")

# Auto-probe: StorageManager tries every registered filesystem's probe()
fs = ctx.storage.auto_probe(layer)
for entry in fs.ls("/"):
    print(entry.name, entry.size)
```

CLI equivalent:

```bash
deepview storage info /path/to/disk.img
deepview filesystem ls /path/to/disk.img /
```

### 7.2 Unlock a LUKS volume with a memory-extracted key

```python
from deepview.storage.containers import MasterKey, Passphrase

# Path 1: key already extracted from a memory dump
result = await ctx.unlocker.auto_unlock(
    layer,
    key_sources=[MasterKey(b"\x00" * 64)],
)

# Path 2: passphrase routed through the offload engine's KDF backend
result = await ctx.unlocker.auto_unlock(
    layer,
    key_sources=[Passphrase("correct horse battery staple")],
)

if result.ok:
    decrypted_layer = result.decrypted_layer
```

CLI equivalent:

```bash
deepview unlock auto /path/to/volume.img \
    --passphrase-env DEEPVIEW_PASSPHRASE
```

See [guides/unlock-luks-volume](../guides/unlock-luks-volume.md) for
the passphrase + memory-extracted-key walkthrough.

### 7.3 Offload a PBKDF2 iteration job

```python
from deepview.offload.kdf import pbkdf2_sha256

future = ctx.offload.submit(
    pbkdf2_sha256,
    password=b"correct horse battery staple",
    salt=b"...",
    iterations=600_000,
    dk_len=32,
    backend="process",          # default; also "thread", "gpu-opencl", "gpu-cuda"
    tag="luks-header-1",
)
result = await future
print(result.backend, result.value.hex())
```

CLI equivalent:

```bash
deepview offload status
deepview offload benchmark --kdf pbkdf2_sha256 --iterations 100000
```

### 7.4 Remote memory acquisition over SSH

!!! warning "Authorization gates"
    Every `remote-image` subcommand requires `--confirm` **and**
    `--authorization-statement`. DMA subcommands additionally require
    `--enable-dma` and a root check, and log an IOMMU warning before
    the first read. Missing gates raise `AuthorizationError`.

```bash
deepview remote-image ssh \
    --host target.example.com \
    --user forensics \
    --key-path ~/.ssh/forensics_id_ed25519 \
    --output /evidence/target.lime \
    --confirm \
    --authorization-statement "Case 2026-04-14-IR-0042, warrant on file"
```

Python API:

```python
from deepview.memory.acquisition.remote import RemoteEndpoint
from deepview.memory.acquisition.remote.factory import build_remote_provider

endpoint = RemoteEndpoint(
    host="target.example.com",
    user="forensics",
    key_path="~/.ssh/forensics_id_ed25519",
)
provider = build_remote_provider("ssh_dd", endpoint)
acquisition = await provider.acquire(output="/evidence/target.lime")
```

See [guides/remote-acquire-ssh](../guides/remote-acquire-ssh.md) and
[guides/remote-acquire-dma](../guides/remote-acquire-dma.md) for the
full gated workflows.

## 8. Behavior changes

None expected to break existing code. The only observable shifts are
the parser bug-fixes called out in sections 4 and 5 — both of which
make previously-failing code paths succeed:

| Area | v0.1 | v0.2 |
|---|---|---|
| Hibernation body reads | Raised (stub) | Decompresses Xpress page runs |
| Crashdump BITMAP_DUMP | Raised (unsupported variant) | Parses cleanly |
| Format detection | Magic-byte only | Magic bytes + extension fallback + MDMP / HIBR / SSM recognition |
| Zero-padded raw | Misclassified as ELF | Classified as `RAW` |

None of these flip a previously-correct answer to a wrong one; they
flip previously-failing answers to correct ones.

## 9. Deprecations

**None.** v0.2 does not deprecate, rename, or warn on any v0.1 surface.
When deprecations arrive in a future minor, they will be announced via
`DeprecationWarning` emitted from the relevant entry point and
documented in the per-version migration page here.

## 10. Reverting

!!! tip "Clean revert path"
    Deep View does not perform on-disk upgrades when you install v0.2
    — no config rewrites, no session-DB migrations, no plugin registry
    changes. Reverting is purely a Python-environment operation.

```bash
# From the repo root
git checkout v0.1.0
pip install -e .

# …or, if you have a separate virtualenv per version
python -m venv .venv-v01
source .venv-v01/bin/activate
pip install "deepview==0.1.0"
```

The only caveat is that reports, session-replay databases, and
exported artifacts produced by v0.2 may reference event classes,
config fields, or `DumpFormat` members that do not exist in v0.1.
v0.1 will raise on deserialising those; v0.2 cleanly reads anything
v0.1 wrote.

!!! warning "Don't mix and match"
    Do not run v0.1 and v0.2 alternately against the same
    `~/.deepview/` state directory if you are using the session replay
    store — replay DBs written by v0.2 may contain new event classes
    that v0.1 cannot reconstruct.
