# ISO/IEC 27037:2012 Mapping

[ISO/IEC 27037:2012](https://www.iso.org/standard/44381.html),
*Information technology — Security techniques — Guidelines for
identification, collection, acquisition and preservation of digital
evidence*, defines the fundamental principles for handling digital
evidence in a way that supports its probative value. This page maps
the four principles and the process model clauses to Deep View
capabilities.

!!! warning "Operational mapping, not a compliance certification"
    ISO/IEC 27037 is a guideline, not a certifiable management system
    — there is no "ISO 27037 certified" programme. ISO 17025
    (laboratory accreditation) and ISO 27001 (ISMS) are the
    certifiable neighbours. Use this page to demonstrate that Deep
    View's design respects 27037's principles; the programme
    documentation is your responsibility.

## Principles (ISO/IEC 27037 §5.3)

ISO/IEC 27037 §5.3 enumerates the four general principles that govern
digital evidence handling:

| # | ISO principle | Summary |
|---|--------------|---------|
| 1 | **Auditability** | Every action taken on the evidence is recoverable — who, what, when, how |
| 2 | **Repeatability** | The same procedure on the same input yields the same output |
| 3 | **Reproducibility** | A different examiner with a different (but equivalent) setup gets the same result |
| 4 | **Justifiability** | The choice of method is defensible and appropriate to the situation |

Deep View's architecture addresses each of these at the subsystem
level. The mapping:

### Principle 1 — Auditability

| ISO §5.3.1 expectation | Deep View realisation | Module |
|------------------------|-----------------------|--------|
| Log every access to the evidence | Typed events on `EventBus` for every acquisition, plugin run, DataLayer open | `src/deepview/core/events.py`, `core/context.py` |
| Identify the operator | `DEEPVIEW_OPERATOR` env var is captured into every event's `actor` field; also mapped to structlog `operator` key | `src/deepview/core/config.py` |
| Timestamp to precision adequate to the task | ISO-8601 UTC with microsecond precision via structlog's `TimeStamper(fmt="iso", utc=True, key="timestamp")` | `src/deepview/core/logging.py` |
| Capture tool version | Deep View version + Python version + platform info recorded in every session's metadata row | `src/deepview/replay/recorder.py::_write_session_metadata` |
| Persist the audit log durably | SessionStore is SQLite with WAL journaling; survives process crash | `src/deepview/replay/store.py` |
| Protect the audit log from tampering | Structlog emits to both the session store *and* an append-only JSONL on the filesystem (`~/.deepview/audit.log` by default); operators are encouraged to mirror to WORM storage | `src/deepview/core/logging.py` |
| Distinguish actions on evidence from actions on metadata | Separate event classes — `EvidenceReadEvent` (evidence access) vs `PluginInvokedEvent` (metadata / analysis) | `src/deepview/core/events.py` |

The audit trail is built from three concentric layers:

1. **structlog** — application-level JSON log to stdout / file.
2. **EventBus** — in-process pub/sub, consumed by the SessionRecorder.
3. **SessionStore** — durable SQLite archive suitable for forensic retention.

All three share the same actor / timestamp / event-ID schema, so
cross-referencing is trivial.

### Principle 2 — Repeatability

| ISO §5.3.2 expectation | Deep View realisation | Module |
|------------------------|-----------------------|--------|
| Deterministic plugin output | Built-in plugins are pure functions of their inputs (no wall-clock, no PID, no random seed in the output) unless explicitly noted | `src/deepview/plugins/builtin/*.py` |
| Explicit non-determinism boundary | Plugins that sample, randomise, or depend on wall-clock document this in their docstring and surface the seed as a `RunOption` | `src/deepview/interfaces/plugin.py::PluginRequirement` |
| Ordered event replay | `SessionReplayer` re-publishes stored events in their original relative order | `src/deepview/replay/replayer.py` |
| Stable hash of results | `PluginResult.content_hash` is a SHA-256 over the canonical JSON serialisation of the result | `src/deepview/interfaces/plugin.py` |
| Version pinning | Deep View version, optional-extras versions (yara-python, volatility3, etc.) captured in session metadata; re-running in the same versioned venv is byte-reproducible | `src/deepview/cli/commands/doctor.py` |
| Same input, same output (test coverage) | Unit tests under `tests/unit/test_plugins/` assert stable output hashes for each built-in plugin | `tests/unit/test_plugins/` |

Repeatability breaks only when:

- the input itself is live (eBPF trace of a live system) — replay
  records the stream, and *that* recording is repeatable;
- an optional dependency is upgraded (e.g. a new Volatility 3 profile
  added a symbol) — the session metadata captures the exact installed
  versions so the deviation is diagnosable;
- an explicit seed or sampling rate is chosen by the operator — the
  seed is captured in the session.

### Principle 3 — Reproducibility

| ISO §5.3.3 expectation | Deep View realisation |
|------------------------|-----------------------|
| Different examiner, same result | Open-source code + pinned deps + captured session config + captured evidence bytes → any examiner can rebuild the environment from `pip install deepview==<version>` |
| Document the toolchain | `deepview doctor` prints the full environment manifest; the same manifest is embedded in every report |
| Reference data is retrievable | YARA rulesets, classifier rulesets, and STIX mappings ship with the release and are versioned; session metadata captures the checksum of each ruleset loaded |
| Hardware-dependent results documented | Hardware-assisted acquisition (PCILeech, chipsec) captures the bus / controller identifier; a different examiner with equivalent hardware reaches the same image |
| Statistical methods disclosed | `detection/anomaly.py` uses a z-score threshold documented in the module; the threshold is parameterised, not hard-coded |

Reproducibility is *stronger* than repeatability: even with a
different examiner and different hardware, running the same
procedure on the same evidence bytes should yield the same findings.
Deep View's support for this is:

- **deterministic plugin catalog** — every plugin is a class with a
  stable module path and a canonical name;
- **hash-addressable evidence** — reports reference evidence by SHA-256
  rather than by path;
- **captured configuration** — `DeepViewConfig.dump_effective()`
  serialises the merged config tree (file + env overrides) into the
  session so a peer can reconstruct it.

### Principle 4 — Justifiability

| ISO §5.3.4 expectation | Deep View realisation |
|------------------------|-----------------------|
| Method appropriate to the situation | Plugin requirements (`get_requirements()`) advertise prerequisites, so the CLI refuses to run an unsuitable plugin against an unsuitable input |
| Alternative methods documented | Memory analysis offers both Volatility 3 and MemProcFS engines; operator selects and justifies the choice |
| Risk of the method disclosed | Dual-use capabilities (packet mangling, hardware DMA) require explicit opt-in flags and fail-closed defaults |
| Operator's decision recorded | Every CLI invocation's argv is captured into `SessionStartedEvent.argv` |
| Precedence for procedural choices | Operator OPSEC guide ([security/opsec.md](../security/opsec.md)) documents the recommended acquisition order and the rationale |

## Process model (ISO/IEC 27037 §6)

ISO 27037 defines four processes: **identification** (§6.1),
**collection** (§6.2), **acquisition** (§6.3), **preservation** (§6.4).
NIST SP 800-86 collapses the first three into a single Collection
phase; the ISO split is finer-grained.

### §6.1 Identification

| ISO clause | Deep View capability |
|------------|----------------------|
| Identify digital devices and potential evidence | `deepview doctor` inventories the current system's live capabilities; `deepview vm list` enumerates hypervisor-visible guests |
| Prioritise volatile data | Acquisition order enforced by `memory/manager.py::acquire()` — memory before disk when both requested |
| Record identification | `DeviceIdentifiedEvent` (structlog + EventBus) |

### §6.2 Collection

| ISO clause | Deep View capability |
|------------|----------------------|
| Physical collection of devices | Out of scope — operator responsibility |
| Logical collection of live data | `inspect/*.py` on-demand primitives (process bundle, memory peek, file inspect, net inspect) |
| Document scene | Operator uploads scene notes into the session via `deepview session annotate` |

### §6.3 Acquisition

| ISO clause | Deep View capability |
|------------|----------------------|
| Acquire exact copies | Raw imaging via `AcquisitionResult` — byte-for-byte, SHA-256 verified |
| Validate integrity (hash pre / post) | Source hash computed during read; target hash computed post-write; mismatch raises `AcquisitionIntegrityError` |
| Maintain hash chain | `SessionRecorder` stores both hashes in the session |
| Use validated tools | Built-in acquisition providers each carry a `validator` module with a known-good test vector in `tests/unit/test_memory/acquisition/` |

### §6.4 Preservation

| ISO clause | Deep View capability |
|------------|----------------------|
| Preserve integrity over time | Evidence files are never opened with write flags by the core pipeline; DataLayer has no `write()` |
| Storage environment controls | Operator responsibility — the guide recommends WORM storage and hardware write-blockers |
| Custody log | SessionStore is the custody log; exports are PDF-printable |
| Transport | Out of scope |

## Replay as an ISO 27037 enabler

The replay subsystem is the unique ISO 27037 contribution of Deep
View. It satisfies:

- **Auditability** — every live event is captured into the session
  store, so after-the-fact "what did the tool see at 14:22:03?" is
  answerable.
- **Repeatability** — `SessionReplayer` re-emits the captured events
  onto a private `TraceEventBus`, so any classifier / renderer
  downstream sees byte-identical input.
- **Reproducibility** — the session store plus the evidence hash plus
  the versioned dependencies lets a *different* examiner run the same
  analysis and obtain the same classifier output.

See `src/deepview/replay/` for the implementation: `store.py`,
`recorder.py`, `replayer.py`, `reader.py`, `circular_buffer.py`, and
`schema.py`.

## What this mapping does not cover

- **Signatures and seals** — ISO 27037 mentions tamper-evident seals
  on physical devices. Out of scope for software.
- **Transport of evidence** — out of scope.
- **Courtroom presentation** — out of scope.
- **Accreditation of the analysis lab** (ISO 17025) — property of
  the lab, not the tool.

## Role alignment (ISO/IEC 27037 §4)

ISO 27037 §4 defines three evidence-handling roles. Deep View maps
them onto operator personae as follows:

| ISO role | Definition | Deep View mapping |
|----------|------------|-------------------|
| **Digital Evidence First Responder (DEFR)** | Identifies, collects, acquires, preserves at the scene | Operator running `deepview memory acquire` / `deepview inspect` on the live host |
| **Digital Evidence Specialist (DES)** | Examines and analyses in the lab | Operator running `deepview memory plugins`, `deepview detect`, `deepview replay` in a controlled environment |
| **Incident Response Specialist (IRS)** | Coordinates the broader incident response | Consumes Deep View reports / STIX bundles via the `reporting/` subsystem |

The `session.roles: list[str]` field in `SessionStartedEvent` lets
the operator tag which role they're currently performing — useful
for auditors reading the session export.

## Relationship with ISO/IEC 27041, 27042, 27043, 27050

ISO 27037 is part of a family of evidence-handling standards. The
sibling standards cover different lifecycle phases:

| Standard | Scope | Deep View coverage |
|----------|-------|-------------------|
| ISO/IEC 27041 | Guidance on assurance of suitability of methods | Test matrix and module docstrings provide method transparency |
| ISO/IEC 27042 | Analysis and interpretation of digital evidence | `detection/*.py` + classifier rulesets; manual interpretation still required |
| ISO/IEC 27043 | Incident investigation principles and processes | CLI workflow + [NIST SP 800-86 appendix D worked example](nist-sp-800-86.md#appendix-d-workflow-example) |
| ISO/IEC 27050 | Electronic discovery | Out of scope — Deep View is forensic, not eDiscovery |

The mapping here focuses on 27037 because it is the foundation
standard. If your programme references any of 27041–27050, the same
Deep View capabilities (structlog, SessionStore, DataLayer read-only
contract) generally apply.

## Example: auditing a Deep View session

A worked example of how to audit a Deep View session against ISO
27037's four principles.

### Auditability check

```bash
# Enumerate every event in the session, chronologically
sqlite3 session-<id>.sqlite \
  "SELECT timestamp, actor, event_type, evidence_hash
     FROM events
  ORDER BY timestamp ASC;"

# Cross-reference with the filesystem audit log
jq 'select(.session_id == "<id>") | {ts: .timestamp, op: .operator, evt: .event}' \
  ~/.deepview/audit.log
```

Every evidence access has an entry in both sources. A discrepancy
between the SessionStore and the audit log is an integrity signal.

### Repeatability check

```bash
# Re-run the session's analysis plugins against the same image
deepview replay rerun --session <id> --format json --out rerun.json

# Diff against the stored results
diff <(jq -S . original.json) <(jq -S . rerun.json)
```

Expected output: empty diff, modulo the timestamps on the replay
envelope.

### Reproducibility check

```bash
# On a different workstation, with the same Deep View version
pip install deepview==<pinned-version>
deepview session import --from session-<id>.tar.gz
deepview replay rerun --session <id> --out rerun-peer.json

# Compare with the original auditor's results
sha256sum rerun-peer.json rerun.json
```

Expected output: identical SHA-256s (modulo minor serialisation
quirks; the `content_hash` of each finding is the canonical
comparison point).

### Justifiability check

Open the session's metadata:

```bash
deepview session show --session <id> --format json | jq '.metadata.purpose'
```

If `purpose` is empty or vague, flag it as a procedural gap. A
defensible session records the reason for each acquisition and the
method chosen.
